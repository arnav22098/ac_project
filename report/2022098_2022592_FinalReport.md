## Abstract

This project investigates whether machine learning can distinguish a reduced-round block cipher from a random permutation by learning relations between paired plaintexts and paired ciphertexts. The chosen primitive is **Speck32/64**, and the task is formulated as a binary classification problem over examples generated with a fixed plaintext XOR difference. We implemented a complete end-to-end pipeline including dataset generation, four input representations, three neural models, result aggregation, and report-quality plotting. The evaluated models are a **Multi-Layer Perceptron (MLP)**, a **1D Convolutional Neural Network (CNN)**, and a **Siamese network**. Experiments were performed for rounds 3 through 8 and aggregated across two training seeds to improve stability. The final results show that **difference-based representations are clearly the most effective**, with the strongest low-round result produced by **Siamese + delta** and the most stable late-round result produced by **MLP + delta**. Under the chosen success criteria, the learned distinguisher remains effective up to about **5 rounds** and then degrades toward chance.

## 1. Introduction

Classical cryptanalysis often relies on carefully designed relations between differences in plaintexts and the corresponding differences in ciphertexts. Differential cryptanalysis is one of the clearest examples: instead of analyzing isolated encryptions, it studies how structured input differences propagate through the cipher. In recent years, machine learning has been explored as a way of learning such relations automatically rather than deriving them completely by hand.

The central question of this project is therefore:

> Can a neural model learn enough structure from paired plaintext-ciphertext examples to distinguish a reduced-round cipher from a random permutation?

This is exactly the setting described in the assignment. We are not doing full cryptanalysis or key recovery; we are building a **relation-based distinguisher**. If the model can identify a non-random relation in the outputs generated by the cipher, then it is effectively learning signal that should not exist under a uniformly random permutation.

The project was designed to satisfy all required deliverables:

- dataset generation code
- ML model implementation and training code
- experimental results and plots
- a written report

The focus of the implementation is correctness, interpretability, and clear comparison across representations and models.

## 2. Problem Formulation

Let \(F\) be an unknown function that is either:

1. a reduced-round block cipher \(E_k\), or
2. a random permutation \(\pi\).

For a fixed plaintext XOR difference \(\Delta P\), we generate a plaintext pair:

\[
(P, P \oplus \Delta P)
\]

and obtain the corresponding outputs:

\[
(C, C') = (F(P), F(P \oplus \Delta P)).
\]

Each sample is labeled as:

- \(y = 1\) if the pair was generated by the reduced-round cipher
- \(y = 0\) if the pair was generated by the random permutation

The learning task is binary classification:

\[
f_{\theta}(P, P \oplus \Delta P, C, C') \rightarrow \{0,1\}
\]

where the model parameters \(\theta\) are learned from data. A successful distinguisher should output higher scores for cipher-generated examples and lower scores for random-permutation examples.

## 3. Cryptographic Setup

### 3.1 Chosen Primitive

The implementation uses **Speck32/64**, a lightweight ARX block cipher with:

- block size: 32 bits
- key size: 64 bits
- word size: 16 bits

Speck was chosen for three reasons. First, it is simple enough to implement and verify in a course-project setting. Second, its round structure makes it suitable for reduced-round experiments. Third, it is a standard example of a lightweight block cipher, which fits the assignment’s list of suggested primitives.

### 3.2 Random Baseline

The assignment requires the same setup to be used for both the cipher and the random baseline. For the baseline, we implemented a **lazy random permutation** over the queried domain:

- when a plaintext is queried for the first time, it is assigned a fresh unused 32-bit output
- if the same plaintext appears again, it receives the same output

This design preserves permutation behavior on the sampled dataset without requiring storage of the full \(2^{32}\)-element permutation. For the purposes of the distinguisher experiment, that is exactly the comparison we need.

### 3.3 Fixed Difference and Rounds

All experiments use the same plaintext XOR difference:

```text
Delta_P = 0x0040
```

The reduced-round settings tested are:

```text
3, 4, 5, 6, 7, 8 rounds
```

This round sweep allows the project to answer the assignment’s key question: **what is the maximum number of rounds for which the learned distinguisher remains effective?**

For every round count, the dataset generation code performs the following steps:

1. sample random 32-bit plaintexts \(P\)

## 6. Input Representations

The assignment explicitly asks students to experiment with multiple representations and discuss their impact. To address that requirement, four representations were implemented.

### 6.1 `delta`

\[
\Delta C = C \oplus C'
\]

This is the most direct representation for differential-style learning. Instead of giving the model two ciphertexts and asking it to infer the useful comparison on its own, it provides the relation itself. This turned out to be the strongest representation overall.

### 6.2 `concat`

\[
C \Vert C'
\]

This representation preserves the raw pair. It is more general than `delta`, but the model must learn both the comparison rule and the final classification decision. As expected, it performs worse than `delta`.

### 6.3 `joint`

\[
(P, C, P \oplus \Delta P, C')
\]

This representation gives the model both inputs and outputs. In theory, it contains more information. In practice, it creates a much larger feature vector and makes optimization harder. The final results show that simply giving more raw information is not enough if the representation is not aligned with the cryptanalytic relation of interest.

### 6.4 `delta_stats`

This representation augments the raw difference bits with simple handcrafted features:

- total Hamming weight of \(\Delta C\)
- left-half Hamming weight
- right-half Hamming weight
- nibble-level Hamming weights
- left-right weight gap

The purpose of this design is to test whether coarse statistical summaries can help the models when the differential signal becomes weaker. The final experiments show that `delta_stats` is not the best overall representation, but it does improve some mid-round behavior, especially at round 4.

## 7. ML Models and Training

### 7.1 Implemented Models

The assignment requires implementation and comparison of at least three models. The project includes:

- **MLP**
- **CNN**
- **Siamese network**

#### MLP

The MLP takes flattened bit vectors and passes them through fully connected layers with ReLU activations, batch normalization, and dropout. It is a strong baseline for fixed-length binary features.

#### CNN

The CNN treats the bit representation as a one-dimensional signal and applies convolutions to capture local bit patterns and short-range structure. This is especially useful when neighboring bits may contain meaningful relations.

#### Siamese Network

The Siamese model splits the input into two halves, processes them through a shared encoder, and combines the embeddings using direct concatenation, absolute difference, and elementwise product. This architecture is natural for pair-based data and produced the strongest low-round performance.

### 7.2 Training Protocol

All experiments use the same basic training setup:

- optimizer: Adam
- learning rate: `0.001`
- batch size: `256`
- weight decay: `1e-5`
- maximum epochs: `16`
- early stopping patience: `4`
- scheduler: `ReduceLROnPlateau`
- device: CPU

To reduce the chance that the final result is caused by a lucky initialization, each configuration is trained with **two seeds**: `1337` and `2027`. The final tables and plots use the **mean** values across seeds and keep track of standard deviations.

### 7.3 Evaluation Metrics

Three evaluation metrics are emphasized:

- **Accuracy**
- **Balanced accuracy**
- **ROC-AUC**

ROC-AUC is important because it evaluates ranking quality across thresholds. Balanced accuracy is important because a weak model can sometimes achieve misleading raw accuracy by favoring one class. For this project, balanced accuracy and ROC-AUC are the most meaningful success metrics.

To define the **maximum effective round**, a configuration must satisfy:

- mean ROC-AUC \(\ge 0.60\)
- mean balanced accuracy \(\ge 0.60\)
- both seeds must satisfy the thresholds

This makes the final conclusion stricter and more reliable than a single lucky run.

## 9. Discussion and Interpretation

### 9.1 Why `delta` Worked Best

The best-performing representation, `delta`, is the one most closely aligned with the intuition behind differential cryptanalysis. If the cipher induces structured relations between paired plaintexts and paired ciphertexts, then those relations are most naturally expressed in the ciphertext difference. By supplying \(\Delta C\) directly, the model starts from the right relational view of the data instead of having to discover that view from raw pairs.

### 9.2 Why `delta_stats` Helped

The `delta_stats` representation adds lightweight handcrafted features on top of the raw bit pattern. It did not overtake `delta` as the best overall feature set, but it produced the best result at round 4. This suggests that even simple statistics such as Hamming weights can add useful signal when the raw relation is beginning to weaken.

### 9.3 Why `joint` Underperformed

At first glance, `joint` seems like it should perform well because it contains the most information. However, more information does not automatically mean easier learning. In this case, the representation becomes much larger, less focused, and harder to optimize. The models appear to benefit more from structured relational features than from an enlarged raw input.

### 9.4 What the Results Mean Cryptanalytically

The project does **not** claim that Speck32/64 is insecure in general, nor does it claim that no distinguisher exists beyond 5 rounds. Instead, the conclusion is narrower and more precise:

- for the chosen plaintext difference `0x0040`
- for the selected representations and model family
- for the available sample budget
- under the strict two-seed stability rule

the learned distinguisher remains effective up to about **5 rounds** and then loses reliable signal.

That is exactly the kind of conclusion expected from a relation-based neural distinguisher project.
